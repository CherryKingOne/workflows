# Python 模型能力目录规划（面向当前 Next.js + Tauri(Rust) 项目）
#
# 本文档用途：
# 1) 只规划目录结构，不写具体业务实现。
# 2) 让新手同学按目录就能完成：新增模型 / 删除模型 / 新增类型 / 重构迁移。
# 3) 满足“前端调用后端函数，后端再调 Python”的 Tauri 函数式调用模式，不走 HTTP 接口。
#
# 重要约束（你提出的规则，已固化到结构设计中）：
# - 目前类型只有两个：image、video。
# - 后续可能新增类型：例如 chat。
# - 每个类型下的模型文件彼此独立：不共享代码、不共享函数、不互相调用。
# - 模型名就是文件名语义：例如 gpt-5.4 -> gpt_5_4.py。
# - 必须有“类型注册 + 模型注册”，后续新增只改注册表 + 新增对应文件。
# - Python 依赖环境统一由 uv 管理，避免“我电脑能跑、你电脑不能跑”的环境漂移问题。
# - URL、API Key、模型名称等运行配置不写死在 Python 文件里，统一从 SQLite 持久化读取。


==============================
一、tauri:dev 启动链路（你关心“如何通过 tauri:dev 启动”）
==============================

当前项目已经有：
- package.json: `npm run tauri:dev` -> `tauri dev`
- tauri.conf.json: `beforeDevCommand` 会先跑 `npm run dev`

建议新增后的运行方式（逻辑不改框架，只补 Python 运行层）：
1) 前端发起 `invoke('run_model_task', {...})`。
2) Rust Tauri Command 收到请求，读取“类型注册 + 模型注册”。
3) Rust 根据注册表定位到具体 Python 文件路径（例如 python_runtime/types/image/gpt_5_4.py）。
4) Rust 用子进程执行该 Python 文件（传入 JSON 参数）。
5) Python 脚本独立执行后返回结果给 Rust。
6) Rust 再返回给前端。

说明：
- 这样不需要单独再手工启动一个 Python Web 服务。
- `npm run tauri:dev` 启动后，前端和 Rust 已就绪；Python 在“有请求时”由 Rust 拉起。
- 这最符合你当前“前端调后端函数”的架构。
- Python 解释器建议固定为 `python_runtime/.venv`（由 uv 创建），让 dev/build 的执行路径一致。

==============================
三、命名规则（必须严格执行）
==============================

1) 模型文件名规则
- 模型原名中的 `-`、`.` 统一改为 `_`。
- 示例：
  - gpt-5.4      -> gpt_5_4.py
  - runway-gen3  -> runway_gen3.py
  - flux-1.1-pro -> flux_1_1_pro.py

2) 类型文件夹规则
- 类型名小写：image、video、chat。
- 类型名就是路由键（注册表里用同名字符串）。

3) 注册表规则
- `type_registry.py` 只登记类型，不登记模型细节。
- `model_registry.py` 按类型登记模型与文件路径映射。
- 删除模型时，先删注册，再删文件，避免运行时找不到文件。


==============================
四、未来新增类型（以 Chat 为例）
==============================

当你未来要新增 `chat` 类型时，按以下步骤做（新手可照抄）：

步骤 1：创建目录
- 新建 `python_runtime/types/chat/`

步骤 2：添加模型文件
- 例如新增 gpt-5.4：`python_runtime/types/chat/gpt_5_4.py`
- 该文件必须只包含 gpt-5.4 的调用逻辑，不复用 image/video 中任何函数。

步骤 3：类型注册
- 在 `python_runtime/registries/type_registry.py` 中增加 `chat`

步骤 4：模型注册
- 在 `python_runtime/registries/model_registry.py` 中增加：
  - type = chat
  - model = gpt-5.4
  - file = python_runtime/types/chat/gpt_5_4.py

步骤 5：Rust 侧接线
- `src-tauri/src/commands/model_runtime.rs` 不需要按模型写死分支。
- 只需走“读注册表 -> 定位文件 -> 执行”的通用流程即可。


==============================
五、删除/重构操作说明（防踩坑）
==============================

A. 删除某个模型（例如删除 image 下的 flux-1.1-pro）
1) 先删 `model_registry.py` 里的映射。
2) 再删 `python_runtime/types/image/flux_1_1_pro.py`。
3) 执行一次 smoke test，确认不会再被调用。

B. 删除某个类型（例如删除 video）
1) 先删 `type_registry.py` 中的 `video`。
2) 再删 `model_registry.py` 中所有 `video` 模型映射。
3) 最后删除 `python_runtime/types/video/` 目录。
4) 确认前端不再展示该类型入口。

C. 重构某个模型文件名
1) 先改文件名。
2) 立刻同步更新 `model_registry.py` 对应路径。
3) 若忘记改注册表，运行时会报“模型已注册但文件不存在”。


==============================
六、为什么这个结构适合你当前项目
==============================

1) 完全符合“前端调后端函数”的 Tauri 模式
- 前端无需改成 HTTP 调用。
- Rust 继续作为统一后端入口，Python 只是能力执行层。

2) 增删改成本低
- 新增模型 = 新增一个 py 文件 + 注册一条映射。
- 删除模型 = 删除一条映射 + 删除一个 py 文件。
- 不会因为公共工具函数耦合而牵一发动全身。

3) 对新手友好
- 目录天然按“类型 -> 模型文件”理解。
- 注册表是唯一入口，定位问题快。
- 每个模型文件职责单一，阅读负担小。

==============================
八、uv 环境管理与打包约束（重点补充）
==============================

为什么必须用 uv：
1) 可复现：`pyproject.toml + uv.lock` 可以锁定依赖版本，减少环境漂移。
2) 新手友好：一条命令就能把环境装好，不需要手工 pip 一个个装。
3) 打包稳定：CI 和本地都能用 `uv sync --frozen` 得到一致环境，便于后续 tauri 打包。

建议固定流程（写进团队开发规范）：
1) 初始化（第一次）
- 在 `python_runtime/` 下执行：`uv sync`
- 生成 `.venv`，并安装锁定依赖

2) 日常开发
- 新增依赖：`uv add <package>`
- 删除依赖：`uv remove <package>`
- 每次依赖变更后提交 `pyproject.toml` 和 `uv.lock`

3) tauri:dev 运行
- Rust `process_runner.rs` 优先使用 `.venv` 里的 python 可执行文件
- 这样可以保证所有同学执行到的是同一个解释器环境

4) tauri:build 前（本地/CI）
- 在构建前跑：`uv sync --frozen`
- 如果 lock 文件与声明不一致，应让构建直接失败，防止带病打包


==============================
九、SQLite 持久化配置 + API 设置页面联动（重点补充）
==============================

这一块你的目标可以拆成“两个数据源”：
1) 注册表（Registry）：定义“系统支持哪些模型”。
2) SQLite 配置（Config）：定义“每个模型当前使用什么 URL/Key/模型名参数”。

请注意这条原则：
- Python 模型文件只负责执行模型调用，不保存配置。
- URL、API Key、模型名称、是否启用等全部由 SQLite 提供。

建议的数据组织（新手易理解）：

A. 注册表层（代码内静态元数据）
- `type_registry.py`：有哪些类型（image/video/chat...）
- `model_registry.py`：每个类型有哪些模型、对应哪个 py 文件、显示名称
- 这部分决定“前端应该展示哪些模型选项”

B. SQLite 配置层（用户可修改持久化数据）
- 建议在 `model_config_repo.rs` 中维护一张模型配置表（可命名 `model_configs`）
- 核心字段建议：
  - `model_type`：image/video/chat
  - `model_key`：注册表里的模型唯一键（例如 gpt-5.4）
  - `base_url`
  - `api_key`（建议加密存储）
  - `runtime_model_name`（真实调用时的模型名，可与 model_key 相同或不同）
  - `is_enabled`
  - `updated_at`
- 建议唯一约束：`UNIQUE(model_type, model_key)`

C. 前端 API 设置弹窗的展示逻辑（你提到的核心需求）
1) 用户点击“API 设置按钮”。
2) 前端调用 `invoke('get_model_catalog_with_configs', { type })`（示例命令名）。
3) Rust 侧执行：
- 先从注册表拿到“该类型全部已注册模型”（即使未配置也要返回）。
- 再从 SQLite 读取该类型已保存配置。
- 最后做一次合并（左连接思路）：返回给前端。
4) 前端展示：
- 已配置模型：显示当前 URL/Key 状态（Key 建议掩码）。
- 未配置模型：也要显示，但标记为“未配置”，提示用户可立即配置。

这能满足你的目标：
- 一旦注册了新模型，即使还没填 key，也会出现在 API 设置页面，用户知道“系统支持它”。
- 删除模型时，只要从注册表删掉，就不会再显示；再清理 SQLite 历史记录即可。

D. 运行时调用逻辑（run_model_task）
1) 前端传入 `type + model_key + payload`
2) Rust 读取注册表定位 Python 文件
3) Rust 用 `(type, model_key)` 去 SQLite 读取该模型配置
4) Rust 将配置值（url/key/runtime_model_name）注入运行参数
5) Python 脚本用传入变量调用模型（不写死任何密钥）

这样做的好处：
- 安全：敏感信息不写在代码里
- 可维护：新增/删除模型只改注册与配置，不改主流程
- 可扩展：后续加 Chat、重构类型、切换供应商都不会推翻现有结构
