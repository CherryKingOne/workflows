# 画布 JSON 整存整取方案（基于当前项目代码）

## 1. 目标与约束

目标：把一个项目的整张画布（节点、连线、位置、配置、视口等）作为一个完整 JSON 字符串存到 `projects` 表的单个字段中，实现“整张存、整张取”。

约束：
- 不拆 `nodes/edges` 子表。
- 后端只保存和返回 JSON 字符串，不做节点级查询。
- 前端负责把运行时状态序列化/反序列化。

---

## 2. 当前代码现状（与你项目对应）

### 2.1 前端画布状态
- 画布核心状态在 `src/presentation/pages/canvas/components/CanvasBoard.tsx`：
  - `canvasNodes`（`useState<CanvasWorkflowNode[]>`）
  - `canvasEdges`（`useState<Edge[]>`）
  - `viewport`（`useState<Viewport>`）
- 目前这些状态只在前端内存中，未接入项目级持久化。

### 2.2 节点数据可序列化风险
- 节点 `data` 里含有函数回调（例如 `onRequestRemove`、`onRequestGenerateImage` 等），定义在：
  - `src/presentation/pages/canvas/components/canvas-nodes/types.ts`
- 这些函数不能直接做稳定持久化，需要序列化前剥离，加载后再挂回。

### 2.3 Rust 后端项目表结构
- 当前 SQLite `projects` 表创建逻辑在：
  - `src-tauri/src/infrastructure/persistence/sqlite/project_repo.rs`
- 现有字段：`id/name/description/created_at/updated_at`，没有 `workflow_data`。

### 2.4 现有 Tauri 项目命令
- 当前只提供：`get_projects/create_project/update_project/delete_project`
- 文件：
  - `src-tauri/src/commands/project.rs`
  - `src-tauri/src/lib.rs`（`invoke_handler` 注册）

---

## 3. 方案总览（最小可落地）

采用一个“画布快照 JSON 包”模型：

1. 保存：前端将当前画布状态打成 JSON 字符串 -> 调用 `save_project_workflow` -> 后端写入 `projects.workflow_data`。
2. 加载：前端调用 `get_project_workflow` -> 后端返回 JSON 字符串 -> 前端解析并恢复节点、连线、视口。

建议 JSON 包结构（Versioned Envelope）：

```json
{
  "schemaVersion": 1,
  "savedAt": "2026-04-09T10:00:00.000Z",
  "viewport": { "x": -1500, "y": -1200, "zoom": 0.89 },
  "nodes": [],
  "edges": []
}
```

说明：
- `schemaVersion` 用于未来兼容升级。
- `nodes/edges` 按 React Flow 可恢复结构保存，但要去掉函数字段。

## 4. 数据库与后端改造

## 4.1 表结构
在 `projects` 表新增字段：

```sql
ALTER TABLE projects ADD COLUMN workflow_data TEXT;
```

建议：
- 允许 `NULL`（老项目兼容）。
- 读取时 `NULL` 视为“空画布”。

因为当前仓储是 `CREATE TABLE IF NOT EXISTS`，没有迁移系统，建议在 `SqliteProjectRepository::new` 中加“列存在检查 + 补列”逻辑（`PRAGMA table_info(projects)`）。

## 4.2 Domain/Application/Repository（Rust）

建议保持你当前 DDD 分层方式：

1. `domain/project/entities.rs`
- `Project` 增加 `workflow_data: Option<String>`（可选）。

2. `domain/project/repositories.rs`
- 在 `ProjectRepository` trait 增加：
  - `save_workflow_data(&self, project_id: &ProjectId, workflow_data: &str) -> Result<(), String>`
  - `get_workflow_data(&self, project_id: &ProjectId) -> Result<Option<String>, String>`

3. `application/project/services.rs`
- 增加应用服务方法：
  - `save_project_workflow(project_id: String, workflow_data: String) -> Result<(), String>`
  - `get_project_workflow(project_id: String) -> Result<Option<String>, String>`

4. `infrastructure/persistence/sqlite/project_repo.rs`
- 实现上面两个方法。
- `save_workflow_data` 同时更新 `updated_at`。

## 4.3 Tauri Commands

在 `src-tauri/src/commands/project.rs` 新增：
- `save_project_workflow(project_id: String, workflow_data: String, service: State<'_, AppProjectService>) -> Result<(), String>`
- `get_project_workflow(project_id: String, service: State<'_, AppProjectService>) -> Result<Option<String>, String>`

并在 `src-tauri/src/lib.rs` 的 `invoke_handler` 注册这两个命令。

---

## 5. 前端改造（CanvasBoard 为主）

## 5.1 新增“序列化/反序列化”模块
建议新增一个专门模块（避免把转换逻辑塞进 `CanvasBoard.tsx`）：
- `src/presentation/pages/canvas/components/workflowPersistence.ts`

核心函数建议：
1. `buildWorkflowSnapshot(nodes, edges, viewport): WorkflowSnapshot`
- 生成可持久化对象。
- 移除节点 `data` 中函数字段（回调）。

2. `parseWorkflowSnapshot(raw: string): WorkflowSnapshot | null`
- JSON 解析 + `schemaVersion` 校验。

3. `hydrateNodesWithRuntimeHandlers(snapshot.nodes, handlers): CanvasWorkflowNode[]`
- 把 `onRequestRemove` 等运行时函数重新挂到节点 `data`。

> 关键点：
> 当前节点 `data` 混有“持久化字段 + 运行时函数”，必须分层处理。

## 5.2 CanvasBoard 接入点
在 `src/presentation/pages/canvas/components/CanvasBoard.tsx`：

1. 加载流程
- 当 `project?.id.value` 可用时调用 `invoke('get_project_workflow', { projectId })`。
- 如果有字符串：解析 -> 恢复 `canvasNodes/canvasEdges/viewport`。
- 如果无数据：保持空画布默认状态。

2. 保存流程
- 给右上角“保存”按钮绑定 `handleSaveWorkflow`。
- `handleSaveWorkflow`：
  - 从当前 `canvasNodes/canvasEdges/viewport` 构建快照。
  - `JSON.stringify`。
  - 调用 `invoke('save_project_workflow', { projectId, workflowData })`。

3. ID 序列恢复
- 加载后根据节点 ID 重算 `nodeSequenceRef`，避免后续创建节点冲突。

## 6. 建议 DTO 契约

前端 TS：

```ts
interface WorkflowSnapshotV1 {
  schemaVersion: 1;
  savedAt: string;
  viewport: { x: number; y: number; zoom: number };
  nodes: Array<Record<string, unknown>>;
  edges: Array<Record<string, unknown>>;
}
```

Tauri 命令层：
- 输入：`projectId: string, workflowData: string`
- 输出：
  - 保存：`Result<(), String>`
  - 读取：`Result<Option<String>, String>`

## 7. 兼容性与容错

1. 老项目兼容
- `workflow_data` 为空时返回空画布。

2. 解析失败
- 如果 JSON 损坏：前端提示“工作流数据损坏，已回退为空画布”。
- 不应导致整个 Canvas 页崩溃。

3. 版本兼容
- `schemaVersion !== 1` 时走兼容分支（先提示不兼容，暂不自动迁移）。

---

## 8. 性能与边界

你的方案本身是“实现快、查询弱”的典型 JSON Blob 方案，适合当前阶段。

建议增加两个软保护：
- 保存前统计字符串大小，超过阈值（如 1MB）给出提醒。
- 超过硬阈值（如 5MB）可拒绝保存并提示用户拆分项目。

后续可选优化（不影响当前方案成立）：
- 压缩（gzip/zstd）后再存。
- 分段快照或增量快照。

## 9. 分步实施计划

Phase 1（MVP，先跑通）：
1. SQLite 增加 `workflow_data` 列。
2. Rust 项目仓储 + service + command 增加 save/load workflow 两个能力。
3. 前端 `CanvasBoard` 接入“保存按钮 -> 保存命令”。
4. 页面初始化时加载并恢复画布。

Phase 2（稳定性）：
1. 增加快照 `schemaVersion`。
2. 增加解析失败兜底和用户提示。
3. 增加大小阈值提示。

Phase 3（可维护性）：
1. 提炼 `workflowPersistence.ts`。
2. 加入 Rust 仓储单测（保存/读取 workflow_data）。
3. 加入前端快照序列化/反序列化测试（如后续引入测试框架）。

---

## 10. 这个方案与需求的匹配结论

完全匹配你的要求：
- 保存：整包 JSON 字符串写入 `projects.workflow_data`。
- 加载：整包字符串读出后前端直接恢复。
- 不做节点级查询。
- 实现复杂度低，数据库改动最小。

代价也明确：
- 不能做“按节点维度检索”类 SQL 查询。
- 大画布时单字段读写成本会上升。

在你当前项目阶段（画布功能快速迭代、后端命令较轻）这是一个合理且落地快的方案。
