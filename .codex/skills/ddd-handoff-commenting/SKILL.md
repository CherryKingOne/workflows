# DDD Handoff Commenting Skill

## 适用场景

当用户提出以下需求时触发本技能：

- 只补充注释，不修改业务逻辑与页面功能
- 需要给新手交接，注释要详细、易懂、可持续维护
- 前端、后端都按 DDD 思路组织
- 后端优先写可复用“功能函数”，不是先设计 HTTP 接口
- 前端通过函数调用后端能力，而不是通过 `fetch/axios/http` 走接口

---

## 核心目标

1. 只做注释层增强，不改现有功能行为。
2. 用注释明确分层职责，降低新同学理解成本。
3. 为后续新增功能提前写“注释预留位”，避免未来继续堆在单文件。
4. 让前后端都具备可扩展、可维护、可交接的结构约束。

---

## 总体架构原则（前后端统一）

- 遵循 DDD：`domain`、`application`、`presentation/infrastructure` 职责清晰。
- 路由层/入口层保持“薄”，只做接线和编排，不堆积细节实现。
- 可独立理解的功能块要拆模块，主文件只负责调用与组合。
- 注释必须解释“为什么这样设计”，不只描述“这段代码做了什么”。

---

## 后端原则（函数能力优先）

### 设计要求

- 后端优先沉淀“功能函数/用例函数”，不是先暴露 HTTP 接口。
- 一个函数只承担一个明确业务能力，输入输出稳定、可测试。
- 业务规则写在 `domain`，编排写在 `application`，技术细节放 `infrastructure`。

### 调用要求

- 前端调用后端能力应优先采用“函数调用”方式。
- 避免新增 `REST/HTTP` 风格接口作为默认路径。
- 若存在框架内置调用机制（例如同仓 Server Function/Action），优先采用函数式调用模型。

### 后端注释重点

- 该函数属于哪一层（domain/application/infrastructure）
- 该函数解决什么业务问题
- 输入参数的业务语义与边界
- 输出结果给前端的契约含义
- 后续可扩展点（新增规则、策略、校验、持久化方式）

---

## 前端原则（调用能力而非堆实现）

### 设计要求

- 页面文件负责“页面入口 + 数据接线 + 状态兜底”。
- 复杂弹窗、卡片、面板、画布节点等应拆分为独立模块。
- 不在单个页面中把所有未来设计一次性写死。

### 调用要求

- 前端优先调用后端功能函数，不默认写 HTTP 请求层。
- 页面/组件聚焦展示与交互，不承担复杂业务规则。
- 通过注释清楚说明“该页面是装配层，不是全部实现层”。

### 前端注释重点

- 当前页面职责边界（做什么、不做什么）
- 为什么要拆模块（交接效率、可维护性、可测试性）
- 新功能应新增到哪里（推荐目录与模块命名）
- 预留功能占位（弹窗、卡片、面板、操作区、扩展点）

---

## 推荐目录规划（示例）

以下是建议，不要求一模一样，但必须体现 DDD 分层与可维护性。

```txt
src/
  domain/
    <bounded-context>/
      entities/
      valueObjects/
      services/
      repositories/
  application/
    <bounded-context>/
      commands/
      queries/
      useCases/
  infrastructure/
    persistence/
    adapters/
  presentation/
    pages/
    components/
    hooks/
```

前端画布类页面可进一步拆分示例：

```txt
src/presentation/pages/canvas/
  components/
    modals/
    cards/
    panels/
    context-menu/
    canvas-nodes/
  hooks/
```

## 注释编写规范（必须遵守）

1. 只写注释，不改逻辑。
2. 注释先讲职责和边界，再讲实现细节。
3. 注释必须能让新手快速回答：
   - 这个文件为什么存在
   - 这个文件应该改什么、不该改什么
   - 新增功能该加到哪里
4. 对未来功能必须添加“预占位注释”，包括新增方向与拆分建议。
5. 避免空话注释，如“设置变量值”；要写设计意图和扩展策略。
6. 术语统一：路由层、展示层、应用层、领域层。

---

## 标准输出要求（执行本技能时）

- 仅修改目标文件中的注释。
- 不新增接口、不改函数签名、不改业务逻辑。
- 交付时说明：
  - 注释覆盖了哪些职责边界
  - 哪些位置写了未来功能预留
  - 是否严格保持“只注释变更”

---

## 禁止事项

- 禁止把“后续可扩展说明”只停留在口头，不落在代码注释里。
- 禁止把所有新需求继续堆到单一页面文件。
- 禁止默认新增 HTTP API 作为前后端协作主路径。
- 禁止写只对作者自己有意义、对新手无帮助的注释。
