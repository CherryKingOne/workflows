### 升级版需求说明：工作区与项目制管理模块

#### 一、 核心概念梳理（研发必须明确的前提）
我们需要区分**“全局配置”**和**“项目独立配置”**。
为了实现“用户直接打包整个项目文件夹发送给同事即可无缝导入”的目标，所有的项目相关产物（工作流数据、下载物料、自动保存）必须做到**高内聚（全部塞在一个文件夹里）**。

#### 二、 目录结构设计规范 (Directory Architecture)

**1. 全局工作区根目录 (Global Workspace)**
*   **定位**：软件所有项目的“家”。这是用户在设置页里唯一需要管理的根目录。
*   **默认路径建议**：不建议放 `Downloads`（因为用户经常一键清空下载文件夹），建议放在 **`Documents (文档) **（使用全英文，避免由于系统语言不同导致路径乱码）。
*   **前端设置页操作**：用户在设置里修改的“默认下载/存储位置”，实际上修改的是这个 `Workspace` 的位置（比如用户可以把它整个移到 D 盘：`D:\ABC_Workspace`）。

**2. 项目级目录结构 (Project Bundle) —— 动态生成**
当用户新建了一个名为 `我爱你` 的项目时，后端需要在工作区下自动生成以下层级：
```text
📦 D:\Documents\          <-- 全局工作区 (设置页配置的根目录)
 ┣ 📂 我爱你/                 <-- 具体的项目文件夹 (按项目命名)
 ┃ ┣ 📜 workflow.json         <-- 【核心】工作流持久化文件 (替代数据库)
 ┃ ┣ 📂 downloads/            <-- 此项目专属的下载文件/物料
 ┃ ┣ 📂 autosaves/            <-- 此项目的自动保存备份 (存放历史版本json)
 ┃ ┗ 📂 cache/                <-- 此项目专属的临时缓存
 ┣ 📂 另一个项目名称/          <-- 其他项目...
```

#### 三、 核心研发需求（需要后端实现的功能）

基于上述结构，后端需要配合实现以下 API 和底层逻辑：

**1. 项目初始化 (createProject)**
*   前端传入项目名称（如 `我爱你`）。
*   后端在配置的全局工作区目录下，创建对应的文件夹结构（包含上述的子文件夹）。
*   同时生成一个初始化的 `workflow.json` 文件。

**2. 纯文件系统驱动的持久化 (JSON Persistence)**
*   **不要用 SQLite 存工作流节点数据！**
*   **保存逻辑**：前端触发“保存/自动保存”时，把整个工作流的 JSON String 丢给后端，后端直接覆写覆盖到该项目目录下的 `workflow.json` 文件中。
*   **自动保存逻辑**：每隔一定时间，后端在 `/autosaves` 目录下写入带有时间戳的文件，例如 `workflow_20231024_1530.json`。
*   **优势（后端会非常赞同）**：I/O 速度极快，免去了繁琐的 DB 增删改查拼接；排查 bug 时，直接打开 json 文本就能看。

**3. 项目导出/打包功能 (exportProject)**
*   前端点击“分享/导出项目”。
*   后端调用系统压缩算法，直接把 `D:\Documents\我爱你\` 这个文件夹打成一个 `.zip` 压缩包（或者你可以自定义后缀名，比如叫 `.abcproj`，本质上还是 zip）。

**4. 项目导入功能 (importProject)**
*   前端点击“导入项目”，调起系统文件选择器，用户选择同事发来的 `.zip` 或 `.abcproj` 文件。
*   后端将其解压到全局 `Documents` 目录下，并读取里面的 `workflow.json` 返回给前端渲染。

### 总结对你 UI 设计的影响：

如果采用了这套高级方案，你的“本地存储管理”弹窗其实会被大幅**简化**：

你不再需要让用户去分别设置“下载目录”、“缓存目录”、“自动保存目录”了（太繁琐了）。
你只需要给用户一个设置项：
*   **工作区位置 (Workspace Location)**：`D:\Documents` `[更改目录]`

剩下的，所有的下载、缓存、自动保存，你的程序都会**自动、默默地**在对应项目的子文件夹里安排好。这就是最顶级的专业软件设计理念（Less is More，用结构规避繁琐的配置）。
