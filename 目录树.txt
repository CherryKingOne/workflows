在当前的 Next.js + Tauri 的架构下，并且要求**完全不使用 Rust**，而是纯粹使用 TypeScript 在前端环境（或者通过 Next.js 的 Node 环境）调用模型，结合你非常明确的 **DDD（领域驱动设计）** 和 **极低耦合（解耦参数与实现）** 的需求，我们可以非常优雅地设计这部分架构。

以下是针对你需求的详细规划、目录结构更新以及代码实现分析。

### 二、 核心设计理念：基于“注册表”的极低耦合架构

根据你的需求：**“独立的参数不可以封装为公共的”、“删除模型只需要删除文件并在注册表中剔除”**。这就是典型的**策略模式（Strategy Pattern）**结合**注册表模式（Registry Pattern）**。

#### 1. 领域层 (Domain) - 定义规则，不关心实现
在领域层，我们只定义所有图像模型必须遵守的“最低契约”，以保证应用层可以统一调用它们，但不限制它们的内部参数。

```typescript
// src/domain/aiModel/services/IImageModelProvider.ts
export interface IImageModelProvider {
  // 返回模型的唯一标识
  getModelId(): string;
  // 统一的执行入口。由于参数互不相通，这里接收一个泛型 Record 或 unknown，由各自模型内部去校验
  generateImage(prompt: string, specificParams: unknown): Promise<string>; // 返回图片URL或Base64
}

// src/domain/aiModel/repositories/IModelRegistry.ts
import { IImageModelProvider } from '../services/IImageModelProvider';

export interface IModelRegistry {
  register(provider: IImageModelProvider): void;
  getProvider(modelId: string): IImageModelProvider;
  removeProvider(modelId: string): void;
}
```

#### 2. 基础设施层 (Infrastructure) - 极低耦合的实际调用代码
这里是你要求的核心：**`Image` 文件夹下的独立模型文件**。

**模型A：DallE3.ts**
```typescript
// src/infrastructure/aiModel/providers/Image/DallE3.ts
import { IImageModelProvider } from '@/domain/aiModel/services/IImageModelProvider';

// 【完全独立】的参数定义，不与任何其他文件共享
interface DallE3Params {
  width: number;
  height: number;
  quality: 'standard' | 'hd';
  apiKey: string; // 后端配置好的Key
}

export class DallE3Provider implements IImageModelProvider {
  getModelId(): string {
    return 'DallE-3';
  }

  async generateImage(prompt: string, params: unknown): Promise<string> {
    // 1. 类型断言与校验（仅限自己文件内部）
    const typedParams = params as DallE3Params;
    
    // 2. 独立的网络请求逻辑 (Fetch)
    const response = await fetch('https://api.openai.com/v1/images/generations', {
      method: 'POST',
      headers: {
        'Content-Type': 'application/json',
        'Authorization': `Bearer ${typedParams.apiKey}`
      },
      body: JSON.stringify({
        model: "dall-e-3",
        prompt: prompt,
        size: `${typedParams.width}x${typedParams.height}`,
        quality: typedParams.quality
      })
    });

    const data = await response.json();
    return data.data[0].url;
  }
}
```

**模型B：StableDiffusion.ts（假设另一个模型）**
```typescript
// src/infrastructure/aiModel/providers/Image/StableDiffusion.ts
import { IImageModelProvider } from '@/domain/aiModel/services/IImageModelProvider';

// 【完全独立】即使也有 width 和 height，也必须在这里重新手写一遍！绝不继承公共接口！
interface SDParams {
  width: number;
  height: number;
  cfg_scale: number;
  apiKey: string;
}

export class StableDiffusionProvider implements IImageModelProvider {
  getModelId(): string {
    return 'SD-XL';

  async generateImage(prompt: string, params: unknown): Promise<string> {

    // 属于 SD 自己的 Fetch 请求逻辑...

```

**注册表实现：LocalModelRegistry.ts**
```typescript
// src/infrastructure/aiModel/registry/LocalModelRegistry.ts
import { IModelRegistry } from '@/domain/aiModel/repositories/IModelRegistry';
import { IImageModelProvider } from '@/domain/aiModel/services/IImageModelProvider';

// 手动引入模型文件
import { DallE3Provider } from '../providers/Image/DallE3';
import { StableDiffusionProvider } from '../providers/Image/StableDiffusion';

export class LocalModelRegistry implements IModelRegistry {
  private providers: Map<string, IImageModelProvider> = new Map();

  constructor() {
    // 【注册模型】：增加或删除模型，只需要改这里和对应文件夹即可
    this.register(new DallE3Provider());
    this.register(new StableDiffusionProvider());
  }

  register(provider: IImageModelProvider): void {
    this.providers.set(provider.getModelId(), provider);
  }

  getProvider(modelId: string): IImageModelProvider {
    const provider = this.providers.get(modelId);
    if (!provider) throw new Error(`Model ${modelId} not found in registry.`);
    return provider;
  }

  removeProvider(modelId: string): void {
    this.providers.delete(modelId);
  }
}
```

### 三、 前端如何调用这些代码？（Application & Presentation）

因为你希望表现为“前端调用后端配置好的模型”，所以 `Application` 层扮演了非常重要的调度角色。

#### 1. 应用服务层 (Application Service)
前端不需要知道模型是怎么实现的，前端只需要发出 Command。

```typescript
// src/application/aiModel/AiModelApplicationService.ts
import { IModelRegistry } from '@/domain/aiModel/repositories/IModelRegistry';
import { InvokeImageGenerationCommand } from './commands/InvokeImageGenerationCommand';
import { LocalModelRegistry } from '@/infrastructure/aiModel/registry/LocalModelRegistry';

export class AiModelApplicationService {
  private registry: IModelRegistry;

  constructor() {
    // 实例化注册表
    this.registry = new LocalModelRegistry();
  }

  async invokeImageModel(command: InvokeImageGenerationCommand): Promise<string> {
    // 1. 从注册表获取指定模型
    const provider = this.registry.getProvider(command.modelId);
    
    // 2. 执行模型（将UI层传来的参数透传给底层独立模型去处理）
    const resultUrl = await provider.generateImage(command.prompt, command.modelParams);
    
    return resultUrl;
  }
}

// 单例导出供前端Hook使用
export const aiModelService = new AiModelApplicationService();
```

#### 2. 表示层 (Presentation / UI)
在 `src/presentation/hooks/useAiModel.ts` 中封装 React Hook：

```typescript
// src/presentation/hooks/useAiModel.ts
import { useState } from 'react';
import { aiModelService } from '@/application/aiModel/AiModelApplicationService';

export const useAiModel = () => {
  const [loading, setLoading] = useState(false);

  const generateImage = async (modelId: string, prompt: string, params: any) => {
    setLoading(true);
    try {
      const url = await aiModelService.invokeImageModel({
        modelId,
        prompt,
        modelParams: params // 这里传入的就是后端预设好的或者UI组装好的参数
      });
      return url;
    } catch (error) {
      console.error(error);
      throw error;
    } finally {
      setLoading(false);
    }
  };

  return { generateImage, loading };
};
```

在你的画布节点（例如 `ImageGenerationNodeCard.tsx`）中直接调用：

```tsx
// src/presentation/pages/canvas/components/canvas-nodes/ImageGenerationNodeCard.tsx
import { useAiModel } from '@/presentation/hooks/useAiModel';

export const ImageGenerationNodeCard = () => {
  const { generateImage, loading } = useAiModel();

  const handleGenerate = async () => {
    // 这里的参数可以是组件内部状态，也可以是你说的“后端配置好的固定参数”
    const config = {
       width: 1024,
       height: 1024,
       quality: 'hd',
       apiKey: 'sk-xxxxxx' // 实际业务中从 ApiConfig 领域获取
    };

    const imageUrl = await generateImage('DallE-3', '一只可爱的猫咪', config);
    console.log("生成的图片:", imageUrl);
  };

  return (
    <div>
      <button onClick={handleGenerate} disabled={loading}>
        {loading ? '生成中...' : '生成图片'}
      </button>
    </div>
  );
}
```

### 四、 为什么要这样设计（呼应你的理念）

你提出的**“极低耦合、甚至容忍代码重复也不做公共封装”**在特定的系统设计中是非常高明的设计准则（这也是微服务架构中常常遵循的 Rule of thumb：“宁可重复，不要错误地耦合”）。

1. **绝对的安全隔离**：如果你修改了 `DallE3.ts` 的逻辑（比如 OpenAI API 升级到了 v2，改了字段名），你完全不需要担心 `StableDiffusion.ts` 会因为继承了同一个父类或者共用了同一个 Interface 而受到影响。
2. **极简的增删流程（符合你的第1和第2点要求）**：
   * **删除模型**：只需要去 `src/infrastructure/aiModel/providers/Image/` 下删掉对应的 `.ts` 文件，并在 `LocalModelRegistry.ts` 中删掉 `this.register()` 这一行。没有任何其他的副作用。
   * **新增模型**：创建一个新文件，实现 `IImageModelProvider`，然后在注册表里 `new` 出来。
3. **符合 DDD 思想（符合你的第3点要求）**：这套方案完全贯彻了 DDD 中**“依赖倒置原则 (DIP)”**。前端 UI (`useAiModel`) 只依赖于 `ApplicationService`。`ApplicationService` 只依赖于 `IModelRegistry` 和 `IImageModelProvider`（都是领域层的抽象）。而具体脏活累活（比如 Fetch API、处理 Header、校验参数）全部被隔离在 Infrastructure 层的边缘文件里。

### 注意事项（Tauri 环境特有）
既然你在前端 TS 里直接发起网络请求调用模型 API：
* 在普通的浏览器环境里会遇到跨域 (CORS) 问题。
* **但是在 Tauri 环境下**，如果你的前端配置允许了外部网络请求，Tauri 实际上是可以通过配置绕过 CORS 的（使用 `@tauri-apps/api/http` 的 `fetch` 代替浏览器的 `window.fetch`）。因此在 `DallE3.ts` 内部，建议根据环境导入 Tauri 的 Http 客户端来发起真实请求。这不会破坏架构，只是底层实现方式的替换。
