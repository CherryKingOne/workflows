文档版本：v1
适用仓库：/Users/edy/Documents/个人项目开发/workflows
适用后端：src-tauri（Rust）
当前迭代范围：仅实现 Image 类型
未来扩展范围：Chat（LLM）、Video 等类型

================================================================
一、文档目标
================================================================

本文件用于指导“模型存储配置”功能开发，重点解决以下问题：
1. 如何在现有项目中按 DDD 分层落地模型配置与模型调用。
2. 如何保证“当前只做 Image”，但后续可以平滑增加 Chat、Video。
3. 如何在不走 HTTP 接口的前提下，通过后端函数能力供前端调用。
4. 如何通过“类型目录 + 模型文件 + 注册表”实现可增删、可重构、低耦合管理。
5. 如何编写适合新手接手的高质量注释，确保长期维护不崩。

说明：
- 本文强调函数能力优先，不设计 REST/HTTP API。
- 前端通过函数调用后端能力（Tauri 命令仅作为函数调用入口，不是 HTTP 接口层）。
- 每个模型调用代码必须独立文件维护，不做“中间公共请求封装”来强耦合模型。


================================================================
二、DDD 分层原则（本功能必须遵守）
================================================================

1. Domain（领域层）
- 放业务概念与规则，不放外部 API 调用细节。
- 定义：模型类型、模型标识、注册信息等核心概念。
- 定义：仓储/注册表抽象（trait），不关心具体存储技术。

2. Application（应用层）
- 编排用例（新增模型注册、删除注册、查询注册、调用模型）。
- 负责流程组织与参数校验，不写第三方 API 细节。
- 面向“一个函数一个能力”。

3. Infrastructure（基础设施层）
- 放具体实现：注册表落盘实现、第三方模型调用实现。
- 这里才出现按类型拆目录、按模型名拆文件。
- 每个模型文件独立实现调用，不共用中间请求封装。

4. Commands（适配层，Tauri 入口）
- 只做“参数接收 -> 调用应用层函数 -> 返回结果”。
- 保持薄，不堆业务规则。
- 不等于 HTTP 接口；只是前端到后端函数能力的调用桥接。


================================================================
三、当前功能边界
================================================================

当前只开发 Image 类型，要求如下：
1. 页面“模型接口配置”只展示 Image（已调整前端入口）。
2. 后端只注册 Image 类型模型。
3. 目录结构必须先为 Image 落地完整模板，后续按同样模式扩展 Chat/Video。

================================================================
五、命名与注册规则（强约束）
================================================================

1. 类型目录命名规则
- 全小写目录名：image、chat、video。
- 当前必须创建并使用 image 目录。
- 后续新增类型时：先有类型目录，再注册模型。

2. 模型文件命名规则
- 文件名必须由模型名转换为 snake_case。
- 示例：
  - gpt-5.4 -> gpt_5_4.rs
  - qwen-image-edit -> qwen_image_edit.rs
- 一个模型一个文件，独立维护，不得把多个模型揉在同一个文件。

3. 注册表规则
- 注册表按“类型 -> 模型列表”组织。
- 每个注册项至少包含：
  - model_type（Image/Chat/Video）
  - model_name（如 gpt-5.4）
  - runtime_key（定位到具体模型调用实现）
  - enabled（是否启用）
- 注册逻辑集中在 registry 模块，便于新增、删除、禁用、迁移。

4. 独立实现规则（重点）
- 不允许抽取“中间公共网络请求封装”去统一所有模型调用。
- 允许共享“最小契约类型”（如输入输出结构体），但每个模型的请求构造、参数映射、错误处理必须写在自己的模型文件里。
- 删除某个模型文件时，不应影响其他模型文件。


================================================================
六、后端函数能力设计（不走 HTTP）
================================================================

以下函数是后端能力中心，供前端调用：

1. register_model(input) -> Result<ModelRegistration>
- 作用：注册一个模型到指定类型。
- 应用层校验：
  - 类型是否合法
  - 模型名是否重复
  - 对应类型目录/运行时实现是否存在

2. remove_model(model_type, model_name) -> Result<()>
- 作用：从注册表删除模型。
- 删除后不影响其他类型和模型。

3. list_models(filter_type) -> Result<Vec<ModelRegistration>>
- 作用：查询模型注册列表，支持按类型过滤。

4. call_model(input) -> Result<ModelOutput>
- 作用：根据类型 + 模型名路由到对应模型文件执行。
- 路由流程：
  - 应用层根据注册表拿到 runtime_key
  - 调用 infrastructure/runtimes/<type>/<model_file>.rs 中函数

5. enable_model/disable_model(...) -> Result<()>
- 作用：启停模型，不删除配置。

说明：
- 前端调用以上能力时，走 Tauri invoke 只是“函数调用桥”。
- 业务设计上仍是“函数能力”，不是 HTTP API 设计。


================================================================
七、注册表设计建议
================================================================

推荐结构（示意）：
- HashMap<ModelType, Vec<ModelRegistration>>

为什么这样设计：
1. 新增类型：直接新增 key（例如 Chat）。
2. 新增模型：在对应 key 下 append 注册项。
3. 删除模型：只删目标项，不触碰其他类型。
4. 使用 SQLite 持久化存储注册信息，应用层函数签名保持稳定。

注册表示意（概念）：
- Image:
  - qwen-image-edit
  - flux-1-dev
- Chat（未来）:
  - gpt-5.4


================================================================
八、以“新增 Chat:gpt-5.4”为例的标准流程
================================================================

注意：当前迭代不实现 Chat，本节仅作为未来操作模板。

步骤 1：创建类型目录（若不存在）
- src-tauri/src/infrastructure/model/runtimes/chat/

步骤 2：创建模型文件
- src-tauri/src/infrastructure/model/runtimes/chat/gpt_5_4.rs
- 文件内写“独立调用代码”，不要依赖中间公共调用封装。

步骤 3：在 chat/mod.rs 暴露模块
- pub mod gpt_5_4;

步骤 4：注册到注册表
- model_type = Chat
- model_name = gpt-5.4
- runtime_key = chat.gpt_5_4

步骤 5：应用层路由补充分支
- call_model 在 Chat + gpt-5.4 时路由到 gpt_5_4.rs 的函数。

步骤 6：编写注释与测试
- 注释说明：该模型入参/出参语义、错误场景、扩展点。
- 测试至少覆盖：注册成功、重复注册失败、调用路由正确、删除不影响其他模型。


================================================================
九、删除/重构策略（防止牵一发而动全身）
================================================================

1. 删除模型
- 先从注册表移除条目。
- 再删除对应模型文件。
- 确保没有其他模型文件引用它。
- 回归验证 list_models 与 call_model 行为。

2. 删除类型
- 先禁用该类型下全部模型并确认无业务依赖。
- 批量移除注册项。
- 删除类型目录（如 chat/）。
- 删除应用层路由分支。

3. 重构模型文件
- 只改目标模型文件，不改其他模型文件。
- 若调整输入输出结构，必须在注释写“破坏性变更说明”。

十、新手交接注释规范（必须执行）
================================================================

注释目标：让新同学 10 分钟内回答下面三件事：
1. 这个文件属于哪一层，负责什么。
2. 新增模型时应该改哪里，不该改哪里。
3. 删除模型时最小影响路径是什么。

A. 文件头注释模板（Rust）
----------------------------------------------------------------
/// ============================================================================
/// 文件：gpt_5_4.rs
/// 分层：Infrastructure / Model Runtime / Chat
/// 职责：仅负责 gpt-5.4 模型调用，不处理注册、路由、持久化
/// 边界：
/// - 应该做：请求构造、响应解析、模型专属错误映射
/// - 不应该做：跨模型公共封装、注册表写入、业务编排
/// 扩展位：
/// - 后续可在本文件新增该模型的高级参数映射（temperature/top_p 等）
/// ============================================================================

================================================================
十一、与前端的协作约定
================================================================

1. 前端只调用后端函数能力，不直接拼第三方模型请求。
2. 前端不依赖模型文件路径，只依赖“类型 + 模型名 + 参数”契约。
3. 前端“模型接口配置”页面当前只展示 Image；后续开放 Chat/Video 时只需：
- 开启对应类型入口
- 调用 list_models(filter_type)
- 保存配置后调用 register_model


================================================================
十二、本迭代实施清单（Image）
================================================================

必须完成：
1. 建立 model 领域上下文（domain/application/infrastructure/commands）。
2. 建立 infrastructure/model/runtimes/image/ 目录。
3. 为 Image 模型创建独立模型文件（按模型名 snake_case 命名）。
4. 完成 Image 类型注册表。
5. 打通 register_model/list_models/call_model/remove_model 函数能力。
6. 每个文件补齐“新手可读”注释（文件头 + 关键函数 + 扩展位说明）。

禁止事项：
1. 禁止把所有模型调用抽成一个公共中间请求器。
2. 禁止把注册逻辑分散到多个不相关模块。
3. 禁止在 commands 层写业务规则。
4. 禁止默认走 HTTP 接口模式。


================================================================
十三、验收标准
================================================================

1. 可扩展性
- 新增一个模型只需新增一个模型文件 + 一条注册，不影响其他模型。

2. 可删除性
- 删除一个模型可做到“注册移除 + 文件删除 + 路由清理”，无连锁破坏。

3. 可读性
- 新手可根据注释独立完成“新增模型/删除模型/定位问题”。

4. 分层正确性
- domain 不含外部调用细节。
- application 只做编排。
- infrastructure 承担具体调用实现。
- commands 保持薄。


================================================================
十四、后续演进建议
================================================================

1. 注册表持久化
- 当前即采用 SQLite 持久化仓储实现，确保重启后注册信息不丢失。

2. 类型开关
- 增加按类型全局启停（如临时关闭 Video 全部模型）。

3. 模型版本管理
- 同模型不同版本可通过 model_name + version 进行细分注册。

4. 观测与排障
- 为每个模型调用增加最小日志字段：model_type、model_name、耗时、错误码。
